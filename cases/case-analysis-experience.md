# 系统架构设计师案例分析备考整理

> 更新时间：2026-04-02  
> 说明：本文基于官方考试规则、近 5 个考试年度的真题入口、以及主流软考平台的案例分析真题整理而成，重点回答四个问题：案例分析是不是上午考、最近 5 年都考了什么、哪些资料值得长期保留、仓库里应该怎么整理这部分内容。文末保留原始链接，方便继续追溯。

## 先回答你的问题

- 是的，除了论文，还有案例分析。
- 按 2024-2025 当前机考常见安排，案例分析在上午，和综合知识连考。
- 但从最近 5 年公开资料看，2021-2023 的公开真题页里仍能看到“下午案例分析”或“下午案例分析和论文”的安排，因此复习时要区分“当前机考安排”和“历年资料口径”。

## 一、案例分析到底什么时候考

工信部发布的软考实施办法明确，高级资格设综合知识、案例分析和论文三个科目。早期规则里，综合知识、案例分析、论文的考试时长分别对应 2.5 小时、1.5 小时和 2 小时。

但从近几年的公开资料看，考试组织方式有过变化：

- 2024 下半年和 2025 年的真题汇总页，都明确写到综合知识和案例分析 2 个科目连考，总时长 240 分钟，时间通常是上午 `8:30-12:30`。
- 但 2023 下半年“下午真题答案及解析”页面仍直接写着“下午考试科目为案例分析和论文”。
- 2022 年和 2021 年的案例分析入口页，也仍按“下午案例分析真题及答案解析”归档。

所以更稳妥的理解应该是：

- 按当前机考常见安排复习：上午综合知识 + 案例分析，下午论文。
- 看历年真题时，不要把不同年份的考试组织方式混成一个口径，要以当年通知和当年真题入口为准。

## 二、案例分析考的不是背诵，而是结构化分析能力

案例分析和论文的差异很明显：

- 论文更像“围绕主题做项目化论述”。
- 案例分析更像“拿到一个具体系统场景后，做拆解、判断、比较和解释”。

从近年资料和真题汇总看，案例分析常见考法通常有这些类型：

### 1. 架构设计与质量属性

- 给定业务系统背景，让你分析高可用、可扩展、性能、安全、可维护等设计点。
- 需要把架构决策和质量属性绑定起来，而不是只写概念定义。

### 2. 建模与设计题

- DFD、ER 图、UML、数据库设计、模块划分这类题目反复出现。
- 这类题很容易丢分，因为很多人知道概念，但一到图和实体关系就写不完整。

### 3. 中间件与分布式场景

- 包括缓存、消息队列、事务一致性、限流降级、服务拆分、分库分表等。
- 题目通常要求你解释原理、指出问题、比较方案或者说明适用场景。

### 4. 新热点与工程化应用

- 最近几年题目明显开始吸收新技术方向，例如知识图谱、区块链、嵌入式 AI、云原生相关能力。
- 这类题不是考你背行业热词，而是看你能不能把新技术放进具体业务场景里分析。

## 三、近年题目方向已经比较清晰

### 2025 上半年方向

根据希赛公开的 2025 上半年案例分析真题与答案，当前能直接从页面文本确认的方向包括：

- 深度学习模型平台下的质量属性场景分析
- 解释器风格架构
- Redis 缓存机制与主从复制
- 医药信息知识图谱与智能问答
- 嵌入式 AI
- 区块链

这组三题很典型：

- 一题偏架构基础与中间件机制
- 一题偏新技术和具体应用场景
- 一题偏质量属性与架构评估方法

### 2025 下半年方向

根据希赛的 2025 下半年真题标签页和考情分析，2025 下半年案例分析继续保持“传统架构基础 + 工程化场景”的组合趋势。站点同时给出了案例分析真题更新入口，说明案例分析仍然是上午重点科目之一。

### 历年高频方向

结合希赛的历年案例分析真题汇总，可以把高频方向先归纳成下面几组：

- 软件架构设计与架构风格
- UML 与结构化分析设计
- 数据库设计与 ER 建模
- Web 与分布式架构
- 可靠性与性能设计
- 信息系统安全设计
- 中间件与平台架构

这意味着案例分析复习不能只靠“刷题数量”，而是要建立题型地图。

## 五、案例分析应该怎么准备

### 1. 先按题型建知识索引

最稳的方法不是先从年份做，而是先按题型做：

- 质量属性类
- 架构风格类
- 数据建模类
- 中间件类
- 分布式与高并发类
- 新技术专题类

这样做的好处是，做完几套题以后你会发现题目在变，但考法没有完全变。

### 2. 每类题都要沉淀“答题模板”

案例分析题最怕写散。建议每类题都积累固定答题模板，例如：

- 若是“优缺点/适用性”题，就按“目标 -> 机制 -> 优点 -> 局限 -> 适用场景”来答。
- 若是“找问题/提改进”题，就按“现象 -> 原因 -> 风险 -> 改进措施 -> 预期收益”来答。
- 若是“比较方案”题，就按“评价维度 -> 方案对比 -> 推荐结论 -> 使用前提”来答。

### 3. 把真题当成专题反查工具

案例分析真题特别适合反推薄弱点：

- 做不出来，不一定是题难，很多时候是某个知识点没有形成结构。
- 一道题背后往往对应一个专题，比如质量属性效用树、消息中间件、数据库建模。
- 所以真题做完以后，应该回到 `notes/` 补专题，而不是只对答案。

### 4. 训练时刻意控制答题粒度

案例分析不是长篇大论，重点是：

- 点到关键机制
- 用术语说清逻辑
- 结构要稳定
- 不写空话

换句话说，案例分析考的是“有限时间内的结构化输出”。

## 六、这个仓库里建议怎么落案例分析

如果这个仓库要服务长期备考，案例分析建议至少沉淀下面几类内容：

| 目录/位置 | 建议内容 |
| --- | --- |
| `cases/` | 真题总览、题型总结、阶段经验贴 |
| `notes/` | 从真题反查出来的专题知识笔记 |
| `review/` | 错题、薄弱专题、答题失误复盘 |

我建议后续在 `cases/` 下面优先补这几类内容：

- 历年案例分析题目清单
- 高频题型模板
- 质量属性类专题题总结
- 数据建模类专题题总结
- 中间件与分布式专题题总结

## 九、最后压缩成一句话

如果论文考的是“项目化表达能力”，那案例分析考的就是“结构化分析能力”。

你后面备考时可以直接这样分工：

- 上午复习优先做案例分析和综合知识联动。
- 下午复习优先做论文主题和素材整理。
- 仓库沉淀上，案例分析负责“题型和方法”，论文负责“素材和表达”。
