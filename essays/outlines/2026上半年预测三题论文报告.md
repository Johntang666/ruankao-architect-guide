### 正文

#### 一、项目背景

我参与的项目是某银行金融市场部资金管理系统及其风控规则引擎 FiboRule。资金管理系统面向交易员、投资经理、风控人员、财务会计和系统管理员，覆盖债券投资、同业拆借、外汇资金、同业存单、现金头寸、非标投资和标准化票据等业务域，采用 qiankun 微前端架构，包含 14 个子应用和 400 多个业务路由，并对接 CFETS 交易中心、中债托管系统、核心账务系统、ESB 总线、大额风险暴露系统、关联交易系统等十余个外部系统。

FiboRule 是资金管理系统的交易风险检测子系统，采用管理端和执行器分离架构。管理端负责指标、规则、评分卡、决策表、决策树、名单库、模型和决策流配置，执行器负责接收资金管理系统的实时风控请求并返回放行、提示、复核或拒绝结果。系统通过指标中心、策略中心、引擎中心和监控中心沉淀风控能力，支持规则动态配置、版本快照和审计追溯。

随着系统运行时间增长，平台积累了大量制度文件、规则版本、指标说明、接口文档、执行日志和审计记录。原有方式主要依赖人工查阅制度和历史配置，存在三个问题：第一，风控制度条款多，业务人员难以快速定位对应规则；第二，新规则配置需要技术人员反复确认指标、阈值和处置动作，效率较低；第三，审计人员追溯某笔交易为什么被拒绝时，需要逐项查看决策流节点和策略命中记录。为提高规则配置和解释效率，我们决定在不改变交易主流程裁决权的前提下，引入基于大模型的智能辅助能力。

#### 二、大模型智能应用架构设计

基于大模型的智能应用不能简单理解为在系统中接一个聊天窗口，而应作为企业应用架构中的一层智能能力。结合银行内网部署、安全审计和风控业务特点，我们将架构设计为六个核心部分。

第一是数据与知识层。该层负责把风控制度、规则配置说明、指标字典、接口文档、历史审计记录和常见问题沉淀为可检索知识。制度文件和接口文档属于非结构化数据，规则、指标、决策流和执行结果属于结构化数据。我们对非结构化文档进行切分、清洗、脱敏和向量化，形成风控知识库；对结构化数据保留原有 MySQL、Redis 和 HBase 存储，通过工具接口按权限查询，避免把敏感交易明细直接暴露给模型。

第二是检索增强层。大模型本身不可靠记忆项目制度和历史配置，因此我们采用 RAG 模式。用户提出“某评级债券是否需要复核”“这条规则应该如何配置”等问题后，系统先根据问题检索相关制度条款、指标定义和历史规则，再把检索结果与问题一起提交给模型。这样既减少模型幻觉，也能让回答引用项目内真实依据。

第三是模型服务层。考虑银行数据安全，我们没有直接把敏感数据发送到公网模型，而是通过模型网关统一接入行内私有化大模型和可替换的外部模型能力。模型网关负责模型路由、提示词模板管理、上下文长度控制、调用限流、敏感词过滤和成本统计。对于制度问答等低风险场景，可以使用通用模型；对于规则配置草案和审计解释，则优先使用行内私有模型。

第四是工具调用层。大模型只负责理解意图和生成建议，真正的数据查询和规则草案生成通过工具完成。我们封装了规则查询、指标查询、决策流查询、结果集查询、规则模板生成和测试样例生成等工具。模型需要查询某条规则时，只能调用受控工具，并由后端根据用户角色校验权限。这样可以避免模型绕过业务系统直接访问数据库。

第五是业务应用层。我们落地了四类智能场景：制度问答、规则配置助手、风控结果解释助手和测试用例生成助手。制度问答帮助业务人员快速定位制度条款；规则配置助手根据制度条款生成规则草案，但草案必须由风控人员审核后才能发布；结果解释助手根据某笔交易的决策流监控和策略命中记录生成可读解释；测试用例助手根据规则条件生成正例、反例和边界值用例。

第六是安全治理层。金融系统不能让大模型直接做最终裁决，因此我们设计了“辅助不裁决”的原则。所有模型输出都带有“建议”属性，不能自动修改生产规则。规则草案进入 FiboRule 后仍需经过测试、审批、版本发布和快照留存。所有模型调用记录用户、时间、输入摘要、检索文档、工具调用和输出结果，供审计追溯。

#### 三、落地过程与关键挑战

架构落地过程中，第一个挑战是知识质量。制度文件和规则配置之间并非一一对应，同一个制度条款可能对应多个指标和规则。为解决这一问题，我们没有只做文档向量化，而是建立了“制度条款、指标、规则、决策流节点、处置动作”的关联关系。风控人员维护规则时，需要选择制度依据；系统将该关联写入元数据表。这样在检索时可以同时返回制度原文和对应规则配置。

第二个挑战是模型幻觉。风控规则涉及交易拦截和合规审计，不能接受模型凭空生成制度依据。我们采取了三项措施：一是回答必须基于检索结果，无法检索到依据时明确返回“未找到制度依据”；二是规则配置草案中必须标注引用的制度条款和指标来源；三是对模型输出增加后处理校验，例如检查指标是否存在、阈值类型是否匹配、处置动作是否在枚举范围内。

第三个挑战是权限与数据安全。资金交易数据和风控结果属于敏感信息，不能让所有用户都能通过自然语言查询。我们将大模型应用接入原有 RBAC 权限体系，用户能看到什么菜单和规则，模型工具就只能查询相同范围的数据。对交易金额、交易对手等敏感字段，在进入模型上下文前进行脱敏；对于需要精确数值判断的场景，则只把计算结果返回给模型，而不暴露明细。

第四个挑战是与既有 FiboRule 架构融合。FiboRule 原本已有指标中心、策略中心、引擎中心和监控中心，我们没有重建一套智能平台，而是在其上增加智能辅助层。规则配置助手调用策略中心模板生成草案，测试用例助手调用规则条件生成样例，结果解释助手读取监控中心的结果集、节点监控和策略监控，再生成业务可理解的解释。这种设计减少了重复建设，也保证智能结果可追溯到原系统数据。

项目上线后，制度检索和规则配置准备效率有明显提升。原来风控人员查找制度依据和历史规则通常需要半天，现在多数问题可以在数分钟内定位。规则草案生成后仍需人工审核，但技术人员不再反复从零解释指标含义和决策流配置方式。审计场景中，系统可以根据结果集自动生成“当时执行哪一版规则、命中了哪些节点、为什么输出复核或拒绝”的解释初稿，提高了审计材料准备效率。

#### 四、效果与反思

基于大模型的智能应用架构给项目带来的最大价值，不是替代风控决策，而是提升知识检索、规则配置和结果解释的效率。项目运行后，风控人员对制度和规则的查询更方便，技术人员在规则配置沟通上的投入减少，审计人员对历史交易的解释材料更一致。更重要的是，系统保留了原有规则引擎的确定性裁决，大模型只承担辅助生成和解释角色，避免了金融业务中过度自动化带来的风险。

当然，该方案也存在不足。大模型输出质量高度依赖知识库质量，制度更新后如果没有及时入库，回答就会滞后。RAG 检索也不能完全解决复杂推理问题，部分跨制度、跨业务的规则仍需人工判断。此外，模型调用增加了系统链路，必须持续关注响应时间、模型成本和上下文安全。后续我们计划进一步完善知识库更新流程，引入规则草案的自动化校验和人工反馈闭环，让模型在实际使用中持续改进。

综上，基于大模型的智能应用架构适合在企业系统中承担知识增强和辅助决策角色。在资金管理系统和 FiboRule 项目中，我们通过知识库、模型网关、工具调用、权限审计和人工复核机制，将大模型能力稳妥地嵌入风控业务流程，在保证安全可控的前提下提升了业务效率。

### 摘要

2023 年，我参与了某银行金融市场部资金管理系统和 FiboRule 风控规则引擎的运行保障改造项目，担任系统架构师。资金管理系统覆盖债券、同业、外汇、同业存单、现金头寸等全业务域，FiboRule 作为其实时风控检测子系统，承接交易录入、审批和确认前的风控调用。随着系统拆分为多个子应用、后端服务、缓存组件和外部接口，传统依靠单机日志排查问题的方式已经难以满足故障定位和性能优化要求。针对交易链路长、外部依赖多、执行器实时性要求高的问题，我主导设计了云原生可观测性架构，通过指标监控、集中日志、分布式链路追踪、告警规则和业务看板，形成从用户请求、交易网关、风控执行器、数据库、Redis 到外部系统的全链路观测能力。项目上线后，故障定位时间明显缩短，性能瓶颈暴露更及时，系统发布和运行保障能力显著提升。

### 正文

#### 一、项目背景

我参与的项目包括某银行金融市场部资金管理系统和 FiboRule 风控规则引擎。资金管理系统采用 qiankun 微前端架构，主应用负责统一登录、菜单、任务列表和跨应用状态，14 个业务子应用分别承载债券、同业、外汇、存单、现金头寸、非标、票据、报表等业务域。系统通过统一交易网关 `/cmsmessage` 接收前端请求，再根据 `prcscd` 处理码路由到具体后端服务，完成经办、审批、确认、记账、打印和报表输出。

FiboRule 风控规则引擎是资金管理系统的重要支撑系统，采用管理端和执行器分离架构。管理端负责指标、规则、策略和决策流配置；执行器负责承接资金管理系统的同步 HTTP 调用，完成规则、评分卡、决策表、名单库、模型和外部接口节点的组合执行。执行器需要在交易高峰期保持 100-300ms 的低延迟响应。

系统上线后，运行保障面临明显挑战。第一，业务链路长，一笔交易从前端页面到交易网关、业务服务、FiboRule 执行器、Redis、MySQL、外部行情和授信接口，任何一环异常都会影响最终体验。第二，日志分散在多个应用和服务器上，故障发生时需要人工登录多台机器检索日志，耗时较长。第三，性能问题不易提前发现，例如 Redis 缓存命中率下降、外部接口延迟升高、数据库慢查询增加，往往要等用户投诉后才被发现。因此，我主导设计了面向云原生运行环境的可观测性架构。

#### 二、可观测性架构设计

云原生可观测性架构的目标，是让系统运行状态可以被度量、追踪、分析和告警。结合项目特点，我们将可观测性分为指标、日志、链路追踪、告警和业务看板五个部分。

第一是指标监控。指标用于回答“系统现在是否健康”。我们把指标分为基础设施指标、应用指标、中间件指标和业务指标。基础设施指标包括 CPU、内存、磁盘、网络和容器实例状态；应用指标包括接口 QPS、响应时间、错误率、线程池队列长度和 JVM GC；中间件指标包括 Redis 命中率、连接数、MySQL 慢查询数、连接池使用率；业务指标包括交易提交量、风控调用量、规则命中率、复核率、拒绝率和报表查询耗时。对于 FiboRule 执行器，我们特别关注 P95 延迟、决策流节点耗时、外部 RPC 节点失败率和规则版本缓存命中率。

第二是集中日志。日志用于回答“系统发生了什么”。我们统一日志格式，每条日志包含 traceId、userId、funcId、prcscd、bktrsq、系统日、服务名、实例号和耗时。前端发起请求时生成或继承 traceId，后端服务、FiboRule 执行器和外部调用都继续传递该 traceId。日志采集后进入集中日志平台，支持按交易流水、用户、处理码和 traceId 检索。对交易金额、交易对手等敏感字段进行脱敏，满足银行审计要求。

第三是分布式链路追踪。链路追踪用于回答“请求慢在哪里”。我们在交易网关、业务服务、FiboRule 执行器、Redis、MySQL 和外部接口调用处埋点，把一次请求拆分成多个 span。对于风控调用链路，系统可以看到规则配置读取耗时、指标获取耗时、每个决策流节点耗时、外部接口耗时和结果写入耗时。当 P95 延迟升高时，架构师可以快速判断瓶颈是在数据库、缓存、规则执行还是外部接口。

第四是告警机制。告警不是简单地把所有异常都发给运维，而是按照严重程度和业务影响分级。比如执行器错误率超过 1% 持续 5 分钟为高优先级告警，Redis 缓存命中率低于 80% 为中优先级告警，单个非核心报表慢查询为低优先级告警。告警规则尽量绑定业务影响，避免告警噪声。对于外部接口异常，系统先判断是否有降级策略，如果熔断已经生效，则告警内容中明确当前降级状态。

第五是业务可观测看板。技术指标对业务人员不直观，因此我们设计了风控运行看板和交易运行看板。风控看板展示当日风控调用量、通过量、提示量、复核量、拒绝量、TOP 命中规则、平均耗时和慢节点排行；交易看板展示各业务域经办、审批、确认和记账量，以及外部系统调用成功率。这样业务、开发、测试和运维可以围绕同一套事实沟通。

#### 三、落地难点与解决措施

第一个难点是跨系统 traceId 贯通。资金管理系统采用微前端，后端又通过统一交易网关和处理码分发，如果 traceId 只在某一层生成，就无法贯穿整个链路。我们在前端 axios 拦截器中生成 traceId，并在请求头传入后端；后端网关如果发现请求没有 traceId，则自动生成；所有内部服务调用和 FiboRule 调用都强制透传。对于外部系统不支持 traceId 的情况，我们在本系统侧记录外部请求流水与 traceId 的映射。

第二个难点是高频监控数据存储压力。FiboRule 每次执行都会产生结果集、节点监控和策略监控，如果全部写入关系型数据库，会影响实时执行性能。我们采用分层存储：实时查询需要的数据写入 MySQL，明细执行日志和节点耗时写入 HBase 或日志平台，热点指标进入时序数据库。执行器先将监控数据写入队列，再由异步线程批量落库，避免阻塞交易主流程。

第三个难点是告警噪声。初期我们把接口错误、慢查询、CPU 高使用率都配置为告警，结果运维人员收到大量低价值通知。后来我们按业务影响重新设计规则，把“单点异常”改为“持续异常 + 影响交易量 + 无自动恢复”的组合条件。例如某外部评级接口偶发超时不立即告警，只有连续失败并导致风控复核率异常升高时才触发高优先级告警。

第四个难点是敏感信息保护。可观测性平台会集中收集大量日志和调用上下文，如果处理不当会形成新的数据泄露风险。我们统一了日志脱敏规则，对交易金额按区间展示，对客户和交易对手名称做掩码，对用户 token、身份证件、账号等字段禁止输出。日志平台查询权限接入原有 RBAC 体系，普通开发人员只能查看技术日志，不能查看敏感业务明细。

#### 四、效果与反思

可观测性架构上线后，系统运行保障能力明显提升。过去定位一次交易提交慢，通常要分别查看前端、交易服务、风控执行器和数据库日志，可能需要一两个小时；现在通过 traceId 可以在链路图中看到每个环节耗时，常见问题能够在十几分钟内定位。性能优化也更有依据，例如我们曾通过慢节点排行发现某个 SQL 指标查询耗时过高，随后增加索引和 Redis 缓存，使该节点平均耗时从 120ms 降到 20ms 左右。

可观测性还帮助我们把系统运行从“事后排查”转向“事前预警”。当 Redis 命中率下降、外部接口错误率升高、执行器线程池队列增长时，系统可以在用户明显感知之前发出预警。业务看板也让风控人员能够看到规则命中分布，及时发现某条规则配置过严或某类交易异常增多。

当然，可观测性建设也有成本。埋点过多会影响性能，日志过细会增加存储成本，告警过多会降低关注度。因此，可观测性不是一次性建设，而是持续治理。后续我们计划进一步引入 SLO 管理，将“风控调用 P95 延迟小于 300ms”“交易提交成功率高于 99.9%”等目标固化为服务等级指标，并与发布准入和容量规划联动。

综上，在资金管理系统和 FiboRule 项目中，云原生可观测性架构通过指标、日志、链路、告警和业务看板，把复杂分布式系统的运行状态显性化。它不仅提升了故障定位效率，也为性能优化、容量规划和业务风险分析提供了可靠依据，是现代企业级系统架构中不可或缺的运行保障能力。

## 三、论零信任安全架构设计及其应用

### 拟题要求

请围绕“零信任安全架构设计及其应用”论题，依次从以下三个方面进行论述：

1. 概要叙述你参与管理和开发的软件项目以及你在其中所担任的主要工作。
2. 详细论述零信任安全架构的核心思想、主要组成和关键机制，并说明其如何保障企业信息系统安全。
3. 结合你具体参与的项目，说明零信任安全架构的选型依据、落地过程中的关键难点及应对措施，以及最终实施效果。

### 摘要

2022 年，我参与了某银行金融市场部资金管理系统及 FiboRule 风控规则引擎的安全架构改造项目，担任系统架构师。资金管理系统覆盖债券、同业、外汇、同业存单、现金头寸等资金业务，涉及交易录入、审批、确认、记账、报表和外部系统交互；FiboRule 则负责规则配置、实时风控检测、版本管理和审计追溯。随着系统拆分为多个前端子应用、后端服务、执行器和外部接口，传统依靠内网边界保护的安全模式已经不足以应对越权访问、接口伪造、脚本注入和敏感数据泄露等风险。针对这些问题，我主导设计了零信任安全架构，以“永不默认信任、持续验证、最小权限”为核心，通过统一身份认证、RBAC 与 ABAC 结合的授权模型、API 网关鉴权、服务间签名、脚本沙箱、数据脱敏和审计追踪等措施，提升系统整体安全性。项目上线后，系统通过银行信息安全审计，未发生重大安全事件。

### 正文

#### 一、项目背景

我参与的项目是某银行金融市场部资金管理系统及其风控规则引擎 FiboRule。资金管理系统面向银行资金业务部门，覆盖债券投资、同业拆借、外汇资金、同业存单、现金头寸、非标投资和标准化票据等业务域。系统采用 qiankun 微前端架构，由主应用统一承载登录、菜单、任务列表和状态广播，多个子应用分别处理不同业务。后端采用统一交易网关和处理码分发模式，通过 `bktrtp`、`funcid`、`taskst` 和 `bktrsq` 串联交易生命周期。

FiboRule 是交易风控检测平台，支持复杂规则、脚本规则、评分卡、决策表、决策树、名单库、模型和 RPC 节点等策略类型。资金管理系统在交易录入、修改、审批和确认前调用 FiboRule，FiboRule 根据已发布的决策流返回放行、提示、复核或拒绝结果。由于该系统涉及交易金额、交易对手、授信额度、风险规则和审计记录，安全要求很高。

在改造前，系统主要依赖银行内网边界、登录 token 和菜单权限控制。但随着微前端拆分、服务接口增多、规则脚本动态化和外部系统集成增加，单纯信任内网和登录态已经不够。我们识别出四类风险：第一，用户登录后可能访问不属于自己角色的业务功能；第二，服务接口可能被伪造请求调用；第三，脚本规则如果缺少限制，可能访问文件、网络或执行危险代码；第四，日志和报表可能泄露交易对手、金额和账户等敏感数据。因此，我主导引入零信任安全架构。

#### 二、零信任安全架构设计

零信任安全架构的核心思想是：不因用户在内网、服务在同一集群或请求来自已登录会话就默认可信，而是在每次访问时根据身份、权限、资源、环境和风险进行持续验证。结合项目特点，我们从身份、设备与环境、访问控制、服务通信、数据保护和审计追踪六个方面设计安全架构。

第一是统一身份认证。用户登录资金管理系统后，认证中心生成包含用户编号、机构、角色、岗位、系统日和过期时间的 token。前端主应用通过路由守卫检查登录态，子应用通过 qiankun 全局状态接收 token，但不能自行生成或篡改身份。后端通过统一过滤器校验 token 签名、过期时间和会话状态。对于长时间未操作或岗位变更的用户，系统强制重新认证。

第二是最小权限授权。项目原有权限以菜单和功能号为主，但零信任要求访问控制更细。我们采用 RBAC 和 ABAC 结合的方式：RBAC 控制用户是否有某类功能权限，例如规则查看、规则修改、规则发布、交易审批、报表导出；ABAC 根据业务属性进一步判断，例如机构、业务线、产品类型、交易金额、流程节点和数据归属。风控人员可以维护规则，但不能直接发布生产版本；交易员可以经办本机构交易，但不能审批自己提交的交易；审计人员可以查看历史结果，但不能修改规则配置。

第三是 API 网关鉴权。所有前端请求统一经过交易网关和 API 过滤器。网关不仅校验 token，还校验 `funcid` 与当前接口是否匹配，防止用户通过修改请求参数访问其他功能。对重要操作，如规则发布、交易审批、记账和报表导出，系统增加二次校验和操作日志。FiboRule 管理端接口通过权限切面统一校验，避免每个业务方法重复编写权限代码。

第四是服务间可信通信。资金管理系统调用 FiboRule 执行器时，不能只依赖内网地址可信。我们在服务间调用中加入应用标识、时间戳、随机数和签名。执行器收到请求后校验调用方是否在白名单中、时间戳是否在允许窗口内、签名是否正确，防止请求重放和接口伪造。对于外部系统调用，通过 ESB 总线统一接入，并按系统维度配置超时、限流和访问白名单。

第五是动态规则的安全隔离。FiboRule 支持 Groovy、Jython 等脚本规则，灵活性高，但也带来脚本注入风险。我们设计了脚本沙箱，限制脚本访问文件系统、网络、系统命令和反射能力，只开放数学计算、字符串处理、日期处理和指定指标读取等白名单 API。脚本执行设置超时时间和内存限制，异常脚本自动终止并记录审计日志。SQL 指标也采用参数化查询、字段白名单和只读账号，防止动态 SQL 注入。

第六是数据保护与审计追踪。交易金额、交易对手、账户、授信额度等字段按敏感等级分类。前端展示、日志输出、报表导出和模型解释都遵循脱敏规则。数据库层面对重要表设置访问控制和备份加密。所有关键操作写入审计日志，包括用户、时间、来源 IP、功能号、操作对象、操作前后值和结果。FiboRule 每次规则发布保存版本快照，每笔交易保存结果集和命中路径，确保能追溯“当时谁配置了规则、哪一版规则被执行、为什么产生该风控结果”。

#### 三、落地难点与解决措施

第一个难点是权限模型细化后带来的复杂度。资金管理业务流程长，涉及经办、审批、确认、记账和报表等多个角色，如果权限设计过细，会增加维护成本。我们采用“角色权限模板 + 业务属性规则”的方式，常见权限通过角色模板配置，特殊限制通过属性规则表达。例如同一交易员不能审批自己提交的交易，不需要为每个人单独配置，而是通过“提交人不等于审批人”的属性规则控制。

第二个难点是旧接口改造。原有接口只校验登录态，没有强制校验功能号与资源归属。我们没有一次性重写所有接口，而是在 API 网关层增加统一校验能力，并优先改造规则发布、交易审批、记账和报表导出等高风险接口。对于低风险查询接口，逐步补充资源归属校验和日志记录，降低改造风险。

第三个难点是安全和性能的平衡。每次访问都进行权限和签名校验，会增加响应时间。我们对用户权限、角色资源关系和服务白名单进行 Redis 缓存，并通过 Canal 监听权限表变更后自动刷新缓存。这样既保证权限变更能及时生效，又避免每个请求都访问数据库。对于执行器实时风控调用，签名校验逻辑保持轻量，只做必要的调用方认证和重放防护。

第四个难点是脚本灵活性与安全边界的冲突。风控人员希望脚本可以表达复杂规则，但安全要求禁止脚本访问危险资源。我们把常用复杂计算沉淀为受控函数库，例如收益率偏离计算、久期校验、集中度计算和评级映射，脚本只能调用这些函数，而不能直接访问系统资源。对于确实需要新增能力的脚本函数，必须经过技术评审后加入白名单。

#### 四、效果与反思

零信任安全架构落地后，系统安全性和可审计性明显增强。用户访问不再只依赖菜单可见性，而是在接口、资源和业务属性层面持续校验。服务间调用通过签名和白名单控制，降低了接口被伪造调用的风险。脚本沙箱和 SQL 防护减少了动态规则带来的注入风险。审计日志和版本快照使关键操作和风控结果都可以追溯，系统顺利通过银行信息安全审计。

该方案也有不足。零信任不是单一技术，而是一组机制的组合，建设成本较高。权限模型越细，配置和测试工作量越大；审计日志越完整，存储和查询成本越高；安全校验越严格，对用户体验和接口性能也会产生影响。因此，零信任架构需要分阶段实施，优先保护高价值资产和高风险操作，再逐步扩展到全系统。

回顾项目实践，我认为零信任安全架构的关键不是简单增加校验，而是改变“内网即可信、登录即可信”的假设。在资金管理系统和 FiboRule 项目中，我们以身份可信、最小权限、持续验证、服务可信、数据保护和全程审计为主线，把安全能力嵌入交易、风控、规则发布和报表导出等核心流程。该架构既满足了金融系统的安全合规要求，也为后续云原生部署、多系统集成和智能化改造奠定了安全基础。
