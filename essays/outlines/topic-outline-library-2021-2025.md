## 2025 年下半年

### 试题一：无服务器架构（Serverless）

**可用素材：** FiboRule 执行器（部分）、资金管理系统报表中心

**提纲：**

1. **摘要**：以资金管理系统为背景，论述在报表查询和批量回测场景中引入 Serverless 架构的设计与实践。
2. **项目背景**：资金管理系统覆盖债券、同业等全业务域，日均数百笔交易。报表中心有 163 个报表路由，部分报表查询和批量风控回测具有明显的波峰波谷特征。
3. **Serverless 架构原理**：按需分配计算资源、自动伸缩、按调用计费、免运维基础设施。
4. **为什么引入 Serverless**：
   - 报表查询在月末/季末出现明显高峰，平时利用率低
   - 批量风控回测需要突发大量计算资源
   - 传统部署需要按峰值预留资源，成本高
5. **落地设计**：
   - 将报表查询、批量回测等非实时任务迁移到 Serverless 函数
   - 交易核心流程仍保持常驻服务，确保低延迟
   - 通过消息队列解耦交易系统和 Serverless 任务
   - 冷启动优化：预热策略、连接池复用、依赖精简
6. **关键挑战**：
   - 冷启动延迟影响用户体验（预热 + 有状态连接池）
   - 数据库连接管理（连接池共享、连接回收）
   - 状态管理（无状态函数 + 外部存储）
   - 与现有微前端架构的集成
7. **效果与反思**：资源利用率提升，运维成本降低；但冷启动和连接管理仍是需要持续优化的点。

**提分点**：区分实时交易（常驻服务）和非实时任务（Serverless）的边界，体现架构取舍。

### 试题二：基于云原生数据库的企业信息系统架构

**可用素材：** 资金管理系统数据库架构、报表与交易库分离

**提纲：**

1. **摘要**：以资金管理系统为背景，论述云原生数据库在企业信息系统中的架构设计与实践。
2. **项目背景**：资金管理系统对接十余个外部系统，数据量持续增长，传统数据库在扩展性、可用性和运维效率上面临挑战。
3. **云原生数据库特征**：存算分离、弹性伸缩、高可用、自动运维、多模数据支持。
4. **数据库架构演进**：
   - 交易库：高并发写入，需要强一致性和低延迟
   - 报表库：复杂查询，需要读写分离和列存加速
   - 缓存层：Redis 缓存热点数据，Canal 同步变更
   - 监控库：高频写入日志，需要时序存储能力
5. **落地设计**：
   - 交易核心使用云原生关系型数据库，存算分离提升扩展性
   - 报表查询使用读写分离 + 列存加速
   - FiboRule 执行器使用 Redis 缓存热点配置，Canal 监听 MySQL 变更同步
   - 监控数据使用 HBase 或时序数据库
6. **关键挑战**：
   - 数据迁移和兼容性（存量数据迁移、SQL 兼容性）
   - 分布式事务一致性（交易与记账的跨库事务）
   - 多数据源路由和连接管理
   - 成本控制和资源规划
7. **效果与反思**：弹性伸缩能力提升，运维自动化程度提高；但分布式事务和数据迁移是主要挑战。

**提分点**：区分不同业务场景对数据库的不同需求（交易 vs 报表 vs 缓存 vs 监控），体现多数据库架构。

### 试题三：系统性能测试技术及其应用

**可用素材：** FiboRule 执行器性能优化、资金管理系统报表性能

**提纲：**

1. **摘要**：以 FiboRule 风控规则引擎为背景，论述系统性能测试的方案设计、实施过程和优化实践。
2. **项目背景**：FiboRule 作为资金管理系统的实时风控检测子系统，需要在交易高峰期间保持低延迟响应。执行器单笔风险检测需要在 100-300ms 内完成。
3. **性能测试方法**：
   - 负载测试：逐步增加并发，观察系统响应时间和吞吐量变化
   - 压力测试：超过预期负载，观察系统极限和降级行为
   - 稳定性测试：持续运行预期负载，观察内存泄漏和资源消耗
   - 基准测试：优化前后的性能对比
4. **测试场景设计**：
   - 单笔决策执行：模拟交易系统调用 `/runner/decision`
   - 批量决策执行：模拟批量回测场景
   - 混合场景：管理端配置查询 + 执行器实时调用并行
   - 外部依赖场景：模拟外部接口延迟和超时
5. **性能瓶颈分析与优化**：
   - 数据库查询慢：SQL 指标查询优化、索引优化、结果缓存
   - 外部接口延迟：超时控制、熔断降级、异步化
   - 规则执行耗时：决策流预编译、节点并行执行
   - 缓存命中率低：缓存预热策略、Canal 同步延迟优化
   - 连接池耗尽：连接池参数调优、连接回收策略
6. **测试工具与监控**：JMeter 压测、APM 链路追踪、Redis 监控、MySQL 慢查询日志
7. **效果与反思**：P95 延迟从 Xms 优化到 Yms；性能测试需要持续进行，不能一次完成。

**提分点**：写出具体的性能指标（P95 延迟、吞吐量、并发数），体现工程真实性。

### 试题四：秒杀场景设计及解决方案

**可用素材：** FiboRule 批量执行、资金管理系统高并发交易

**提纲：**

1. **摘要**：以资金管理系统在月末集中交易和批量风控回测场景为背景，论述高并发场景下的架构设计方案。
2. **项目背景**：资金管理系统在月末、季末出现交易高峰，FiboRule 需要在短时间内处理大量风控检测请求。
3. **秒杀场景特征**：瞬时高并发、读多写少、库存有限、请求可丢弃。
4. **高并发设计**：
   - 限流：网关层限流（令牌桶/漏桶）、应用层限流（信号量/线程池）
   - 削峰：消息队列缓冲请求，异步处理
   - 缓存：Redis 缓存热点数据（规则配置、指标、名单）
   - 熔断：外部接口熔断降级，返回默认结果
   - 异步化：批量执行使用线程池异步处理
5. **FiboRule 的高并发设计**：
   - 执行器无状态设计，支持多实例水平扩展
   - Redis 缓存引擎配置和规则，减少数据库访问
   - 批量执行使用受控线程池并发处理
   - 外部指标接口超时和熔断保护
   - 限流保护：超过承载能力时快速失败
6. **资金管理系统的高并发设计**：
   - 交易提交排队和限流
   - 报表查询异步化和结果缓存
   - 打印任务队列化
7. **效果与反思**：系统承载能力从 X TPS 提升到 Y TPS；但限流策略需要根据业务场景动态调整。

**提分点**：区分"可丢弃请求"（风控提示）和"不可丢弃请求"（交易记账），体现业务判断。

### 试题一：系统负载均衡设计方法

**可用素材：** FiboRule 执行器集群、资金管理系统微前端

**提纲：**

1. **摘要**：以 FiboRule 风控规则引擎为背景，论述系统负载均衡的设计与实践。
2. **项目背景**：FiboRule 采用管理端与执行器分离架构，执行器承接资金管理系统的实时风控调用，需要在交易高峰期间保持稳定。
3. **负载均衡层次**：
   - DNS 负载均衡：多机房部署时的流量分配
   - 网关负载均衡：Nginx/API Gateway 的请求分发
   - 应用层负载均衡：服务间的负载均衡（Ribbon/LoadBalancer）
4. **负载均衡算法**：轮询、加权轮询、最少连接、一致性哈希、IP Hash
5. **落地设计**：
   - 执行器无状态设计，通过 Nginx 做反向代理和负载均衡
   - 管理端和执行器分别部署，各自独立扩容
   - 健康检查：定期检测执行器实例状态
   - 会话保持：风控调用无状态，天然支持负载均衡
6. **关键挑战**：
   - Redis 缓存一致性：多实例间缓存同步
   - Canal 同步延迟：缓存更新的最终一致性
   - 执行器扩缩容：规则版本切换的一致性保障
7. **效果与反思**：执行器支持水平扩展，交易高峰期间自动扩容。

**提分点**：强调无状态设计是负载均衡的前提条件。

### 试题二：事件驱动架构

**可用素材：** FiboRule Canal 同步、资金管理系统工作流状态流转

**提纲：**

1. **摘要**：以资金管理系统和 FiboRule 风控引擎为背景，论述事件驱动架构在金融交易系统中的设计与实践。
2. **项目背景**：资金管理系统交易生命周期涉及经办、审批、确认、记账等多个状态流转；FiboRule 需要监听数据库变更同步缓存。
3. **事件驱动架构原理**：事件生产者、事件通道、事件消费者；松耦合、异步处理、最终一致性。
4. **事件驱动的应用场景**：
   - **场景一：Canal 监听 MySQL 变更**
     - FiboRule 规则配置变更后，Canal 监听 binlog 变更事件
     - 自动同步到 Redis 缓存，减少人工刷新
     - 事件通道：Canal → Redis
   - **场景二：交易状态流转**
     - 经办提交 → 审批事件 → 确认事件 → 记账事件
     - 每个状态变更触发后续动作
     - 事件通道：工作流引擎
   - **场景三：外部系统数据同步**
     - CFETS 下行数据匹配事件
     - 中债托管数据变更事件
     - 事件通道：消息队列
5. **事件驱动的优势**：
   - 系统间松耦合：交易系统不直接依赖缓存刷新逻辑
   - 异步处理：不阻塞主流程
   - 可扩展：新增消费者不影响生产者
6. **关键挑战**：
   - 事件顺序性：Canal 保证 binlog 顺序，但消费端需要幂等处理
   - 事件丢失：消息持久化、消费确认、死信队列
   - 最终一致性：缓存和数据库的短暂不一致窗口
7. **效果与反思**：降低了系统间耦合，提高了可维护性；但事件驱动增加了调试和排障的复杂度。

**提分点**：写出具体的事件流转路径和消费逻辑，不要只讲理论。

### 试题三：多模型数据库及应用

**可用素材：** 资金管理系统多数据源、FiboRule Redis/HBase/MySQL

**提纲：**

1. **摘要**：以资金管理系统和 FiboRule 为背景，论述多模型数据库在金融系统中的架构设计与实践。
2. **项目背景**：金融交易系统数据形态多样：交易数据需要关系型存储、配置缓存需要 KV 存储、监控日志需要列族存储、报表数据需要分析型存储。
3. **多模型数据库选型**：

   | 数据类型 | 数据库 | 场景 |
   |---|---|---|
   | 交易数据 | MySQL | 交易记录、审批流、会计分录 |
   | 配置缓存 | Redis | 规则配置、引擎节点、指标定义 |
   | 监控日志 | HBase | 决策流执行日志、节点监控、策略监控 |
   | 报表数据 | 列存/OLAP | 复杂查询、聚合统计、监管报表 |

4. **数据同步机制**：
   - MySQL → Redis：Canal 监听 binlog 同步
   - MySQL → HBase：监控数据异步写入
   - MySQL → 报表库：ETL 定时同步
5. **数据一致性保障**：
   - 交易数据：强一致性，事务保障
   - 缓存数据：最终一致性，Canal 同步 + 缓存预热
   - 监控数据：异步写入，允许短暂延迟
   - 报表数据：T+1 准实时
6. **关键挑战**：
   - 多数据源路由和连接管理
   - 跨数据源的事务一致性
   - 数据迁移和版本兼容
7. **效果与反思**：针对不同数据形态选择最合适的存储方案；但多数据库增加了运维复杂度。

**提分点**：体现"针对不同数据形态选择不同数据库"的架构判断力。

3. **Lambda 架构**：批处理层（Batch Layer）+ 速度层（Speed Layer）+ 服务层（Serving Layer）
4. **落地设计**：
   - 批处理层：T+1 ETL 同步交易数据到报表库，生成监管报表和经营报表

   - 批处理：规则命中趋势分析、策略效果评估、交易对手风险分布

6. **关键挑战**：批处理和实时结果的一致性、数据延迟窗口、存储成本

**提分点**：区分批处理和实时的不同业务场景，体现架构选择的业务驱动。

### 试题二：模型驱动架构设计方法及其应用

**可用素材：** FiboRule 规则配置与决策流、资金管理系统处理码模型

**提纲：**

1. **摘要**：以 FiboRule 风控规则引擎为背景，论述模型驱动架构在规则引擎设计中的应用。
2. **项目背景**：FiboRule 需要支持风控人员通过可视化界面配置规则，而不是编写代码。
3. **模型驱动架构**：平台无关模型（PIM）→ 平台相关模型（PSM）→ 代码生成
4. **FiboRule 的模型驱动设计**：
   - **PIM 层**：业务人员定义的规则逻辑（条件、动作、输出）
   - **PSM 层**：规则版本表、节点配置表、决策流配置
   - **代码/执行层**：执行器加载配置并递归执行
5. **具体模型**：
   - 指标模型：`t_field` 定义指标的来源、类型、计算公式
   - 规则模型：`t_rule` + `t_rule_version` 定义规则的条件树和版本
   - 引擎模型：`t_engine` + `t_engine_version` + `t_engine_node` 定义决策流
   - 输出模型：`t_resultset` 定义执行结果
6. **可视化决策流编排**：拖拽式节点编排、节点配置面板、连线定义执行顺序
7. **关键挑战**：模型表达能力与执行效率的平衡、模型校验、版本管理
8. **效果与反思**：风控人员可自助配置规则，降低了对技术团队的依赖。

**提分点**：把"规则配置"上升到"模型驱动"的架构层面来论述。

### 试题三：单元测试方法及应用

**可用素材：** FiboRule 规则执行测试、资金管理系统校验逻辑测试

   - 审计安全：操作日志、行为审计、异常检测
4. **FiboRule 的安全设计**：
   - 接口鉴权：签名校验、时间戳防重放、IP 白名单
   - 脚本沙箱：限制 Groovy/Jython 的资源访问和危险操作
   - SQL 注入防护：参数白名单、预编译、只读账号
   - 权限隔离：规则配置、发布、查看分角色控制
5. **资金管理系统的安全设计**：
   - token 路由守卫：未登录跳转登录页
   - 功能号权限：`funcid_` 控制菜单和操作权限
   - 业务解锁：异常状态下的手动恢复机制
   - 前端校验：体验型预校验，强规则后端裁决
6. **关键挑战**：安全性与可用性的平衡、安全策略的持续更新
7. **效果与反思**：建立了多层次的安全防护体系。

### 试题三：企业集成平台的理解与应用

**可用素材：** 资金管理系统 ESB 集成、FiboRule 外部数据源集成

**提纲：**

1. **摘要**：以资金管理系统为背景，论述企业集成平台在金融系统中的设计与实践。
2. **项目背景**：资金管理系统需要对接 CFETS、中债、核心、ESB 等十余个外部系统。
3. **企业集成模式**：
   - 消息模式：消息队列解耦系统间通信
   - 路由模式：基于内容的路由、消息过滤
   - 转换模式：数据格式转换、协议适配
   - 端点模式：服务网关、适配器
4. **资金管理系统的集成架构**：
   - ESB 总线：统一对接外部系统，协议转换、路由、安全控制
   - CFETS/CSTP 集成：下行数据匹配、联机交易
   - 中债/直联集成：交易数据核验、托管账户匹配
   - 核心账务集成：会计分录、记账、核心流水
5. **FiboRule 的集成设计**：
   - SQL 指标：动态数据源访问外部数据库
   - 接口指标：HTTP 调用外部服务
   - RPC 节点：决策流中调用外部服务
6. **关键挑战**：系统可用性、数据一致性、协议差异、版本兼容
7. **效果与反思**：统一集成架构降低了系统间耦合，提高了集成效率。

### 试题四：微服务架构及其应用

**可用素材：** 资金管理系统 qiankun 微前端、FiboRule 服务拆分

**提纲：**

1. **摘要**：以资金管理系统为背景，论述微服务架构在金融交易系统中的设计与实践。
2. **项目背景**：资金管理系统覆盖 14 个业务域，需要支持团队并行开发和独立发布。
3. **微服务拆分原则**：单一职责、业务域边界、数据独立、团队自治。
4. **资金管理系统的微服务拆分**：
   - 前端：qiankun 微前端，14 个子应用独立开发、独立构建
   - 后端：交易域服务、风控服务、会计服务、报表服务、系统服务等
   - FiboRule：管理端与执行器分离，指标中心/策略中心/引擎中心/监控中心模块化
5. **服务治理**：
   - 统一网关：`/cmsmessage + prcscd` 处理码分发
   - 服务间通信：同步 HTTP + 异步消息
   - 配置管理：环境变量 + 配置注入
   - 链路追踪：`bktrsq` 业务流水贯穿
6. **关键挑战**：分布式事务、数据一致性、服务间依赖、版本管理
7. **效果与反思**：团队并行能力提升，发布独立性增强，但运维复杂度增加。
