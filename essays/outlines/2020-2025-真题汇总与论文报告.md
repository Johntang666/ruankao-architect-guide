#### 试题一：论 Serverless 架构模式

> Serverless 架构随着云计算技术的迭代与微服务架构的普及，企业对 IT 系统的弹性伸缩、成本优化及运维效率提出了更高要求——既需快速响应业务峰值需求，又需降低闲置资源消耗，同时减少基础设施运维负担。Serverless 架构模式（无服务器架构）通过"函数即服务（FaaS）"与"后端即服务（BaaS）"的核心理念，实现了"按需分配资源、按使用付费、无需关注底层运维"的核心价值，有效解决了传统架构中资源利用率低、运维成本高的痛点，已广泛应用于 API 服务、事件驱动型应用、定时任务等场景，也是云计算领域架构设计的核心考点之一。

## 三、论文报告：论面向方面的编程技术及其应用（AOP）

> 对应真题：2021年下半年-试题一
> 使用素材：FiboRule 风控规则引擎（主）+ 紫金资金管理系统（辅）

### 摘要

2021年，我参与了某银行金融市场部的资金管理系统风控平台建设项目，担任系统架构师。该项目为资金管理系统的交易风险检测子系统，支持债券、同业存单等交易业务的实时风控决策。针对风控系统中监控、审计、权限等横切关注点散布在多个业务模块、导致代码重复和维护困难的问题，我在项目中引入了面向方面的编程（AOP）技术。通过定义监控切面、审计切面、权限切面和日志切面，将横切关注点从业务代码中剥离，实现了关注点分离和代码复用。项目上线后，横切逻辑的代码重复率降低了约 70%，新增业务模块时无需重复编写监控和审计代码，系统可维护性显著提升。

### 正文

#### 一、项目背景

2021年，我参与了某银行金融市场部的资金管理系统风控平台（FiboRule）建设项目，担任系统架构师。该平台定位为资金管理系统的交易风险检测子系统，覆盖债券投资、同业存单等交易业务。资金管理系统在交易录入、审批、确认前，将交易要素、交易对手、产品、组合、授信、行情等指标传入 FiboRule；FiboRule 根据已发布的决策流执行名单、规则、评分卡、决策表、模型等策略，返回放行、提示、复核或拒绝结果。

项目采用 Spring Boot 2.4.2 框架，包含 839 个 Java 文件，分为指标中心、策略中心、引擎中心、监控中心、分析中心和系统管理六大模块。在项目开发过程中，我们遇到了一个突出问题：监控、审计、权限校验、日志记录等横切关注点散布在各个业务模块中。例如，每个规则执行方法都需要记录执行耗时和结果，每个管理端接口都需要校验用户权限和记录操作日志，每个引擎版本发布都需要记录审批信息。这些横切逻辑如果在每个方法中重复编写，会导致代码冗余、维护困难、修改风险高。

为解决这一问题，我在项目中引入了面向方面的编程（AOP）技术，通过 Spring AOP 框架将横切关注点统一封装为切面，实现了关注点分离。

#### 二、AOP 技术原理与设计

面向方面的编程（AOP）是一种将横切关注点从业务逻辑中分离出来的编程范式。其核心概念包括：

**切面（Aspect）**：横切关注点的模块化封装，包含切入点和通知。在本项目中，我们定义了四个核心切面：监控切面、审计切面、权限切面和日志切面。

**连接点（JoinPoint）**：程序执行过程中的特定点，如方法调用、异常处理等。Spring AOP 中连接点通常指方法执行。

**通知（Advice）**：切面在连接点执行的动作，包括前置通知（Before）、后置通知（After）、返回通知（AfterReturning）、异常通知（AfterThrowing）和环绕通知（Around）。

**切入点（Pointcut）**：匹配连接点的表达式，用于定义通知在哪些方法上执行。我们使用 `@annotation` 和 `execution` 表达式精确匹配需要拦截的方法。

**织入（Weaving）**：将切面应用到目标对象的过程。Spring AOP 使用运行时代理实现织入，编译期无需额外处理。

在架构设计上，我们将 AOP 切面层放在领域服务层之上，切入点覆盖规则执行、指标获取、引擎管理、用户操作等核心方法。切面通过 Spring 的 `@Aspect` 和 `@Component` 注解定义，由 Spring 容器自动代理目标对象。

#### 三、项目落地实施

在 FiboRule 项目中，我们具体实现了四个 AOP 切面：

**监控切面**：用于记录每个节点和策略的执行耗时和结果。切入点定义为 `execution(* com.fiborule.enginex..*.runNode(..))` 和 `execution(* com.fiborule.strategyx..*.execute(..))`。使用环绕通知（Around），在方法执行前记录开始时间，执行后计算耗时，并将监控数据写入 `t_monitor_node` 和 `t_monitor_strategy` 表。对于执行耗时超过阈值的方法，触发慢执行告警。

**审计切面**：用于记录管理端的规则配置、发布、审批等操作。切入点通过自定义注解 `@AuditLog` 标记需要审计的方法。使用后置通知（AfterReturning），记录操作人、操作时间、操作类型、操作对象和操作结果。审计日志写入 `t_logger` 表，支持后续查询和追溯。

**权限切面**：用于校验用户对规则、指标、引擎的操作权限。切入点通过自定义注解 `@RequirePermission` 标记需要权限校验的方法。使用前置通知（Before），在方法执行前校验当前用户是否具有对应资源的操作权限。权限模型基于 RBAC，通过 `t_user_role_rel` 和 `t_role_resource_rel` 表实现用户-角色-资源的权限链。

**日志切面**：用于统一记录接口调用日志。切入点定义为 `execution(* com.fiborule..*.controller.*.*(..))`。使用环绕通知，记录请求参数、响应结果、执行耗时和异常信息。

在实施过程中，我们遇到了几个关键挑战：

第一，切面执行顺序问题。当多个切面同时作用于同一个方法时，需要控制执行顺序。我们通过 `@Order` 注解定义切面优先级：权限切面（Order=1）最先执行，其次是日志切面（Order=2），然后是业务方法，最后是监控切面（Order=3）和审计切面（Order=4）。

第二，切面性能影响问题。监控切面的环绕通知会增加方法执行时间，特别是在高频调用的规则执行方法上。我们通过异步写入监控数据来缓解：监控数据先写入内存队列，由后台线程批量写入数据库，避免阻塞业务方法。

第三，异常处理问题。当业务方法抛出异常时，监控切面需要记录异常信息，审计切面需要记录失败操作。我们使用 `@AfterThrowing` 通知捕获异常，并在监控和审计日志中记录异常类型和堆栈信息。

第四，切入点表达式维护问题。随着模块增加，切入点表达式需要同步更新。我们通过自定义注解（如 `@Monitored`、`@AuditLog`、`@RequirePermission`）标记需要拦截的方法，避免直接依赖包路径和方法名，提高了切入点的可维护性。

#### 四、效果与反思

项目上线后，AOP 技术的应用取得了以下效果：

一是代码重复率显著降低。横切逻辑从业务代码中剥离后，监控、审计、权限、日志等代码统一在切面中维护，业务模块只需添加注解即可获得横切能力，代码重复率降低了约 70%。

二是新增业务模块的开发效率提升。开发人员只需关注业务逻辑，通过注解声明横切需求，无需重复编写监控和审计代码。新增一个策略类型的开发时间从约 3 天缩短到约 1 天。

三是横切逻辑的修改成本降低。例如，当需要修改监控数据的存储方式时，只需修改监控切面一个地方，无需逐个修改数百个业务方法。

四是审计追溯能力增强。通过审计切面，管理端的所有操作都有完整记录，支持审计人员查询"谁在什么时间对什么规则做了什么操作"。

当然，AOP 技术的应用也存在一些不足。一是切面增加了代码的理解成本，新开发人员需要理解 AOP 的代理机制和切入点表达式。二是调试困难，当切面逻辑出现 Bug 时，断点调试不如直接代码直观。三是 Spring AOP 基于代理模式，对同类方法调用（this 调用）无法拦截，需要通过 `AopContext.currentProxy()` 解决。

回顾整个项目，AOP 技术在金融风控系统中的应用是成功的。它有效解决了横切关注点的代码重复和维护困难问题，提高了系统的可维护性和开发效率。在未来的系统演进中，我们计划将更多横切逻辑（如缓存管理、限流控制、参数校验）纳入切面管理，进一步提升架构的整洁度。

## 四、论文报告：论事件驱动架构

> 对应真题：2025年上半年-试题二
> 使用素材：FiboRule Canal 同步（主）+ 紫金资金管理系统工作流状态流转（辅）

### 摘要

2021年，我参与了某银行金融市场部的资金管理系统及其风控平台（FiboRule）建设项目，担任系统架构师。针对风控规则配置变更后缓存同步延迟、交易系统与风控系统间耦合度高的问题，我在项目中引入了事件驱动架构。通过 Canal 监听 MySQL binlog 变更事件自动同步 Redis 缓存，通过工作流状态机驱动交易生命周期流转，通过消息队列解耦外部系统数据同步。项目上线后，规则配置变更到缓存生效的延迟从分钟级降低到秒级，系统间耦合度显著降低，可维护性提升。

### 正文

#### 一、项目背景

2021年，我参与了某银行金融市场部的资金管理系统及其风控平台（FiboRule）建设项目，担任系统架构师。资金管理系统覆盖债券、同业、外汇、存单等全业务域，FiboRule 作为其交易风险检测子系统，支持规则动态配置、实时检测和审计追溯。

在项目运行过程中，我们遇到了两个突出问题。一是 FiboRule 的规则配置变更后，需要手动刷新 Redis 缓存才能生效，操作繁琐且容易遗漏，导致执行器读取到旧规则配置。二是资金管理系统与 FiboRule 之间采用同步 HTTP 调用，交易系统直接依赖风控系统的可用性，耦合度高。

为解决这些问题，我在项目中引入了事件驱动架构，通过事件生产者、事件通道和事件消费者的模式，实现系统间的松耦合和异步处理。

#### 二、事件驱动架构原理与设计

事件驱动架构（EDA）是一种以事件为核心的软件架构模式。其核心组成包括：

**事件生产者**：产生事件的组件。在本项目中，MySQL 数据库的 binlog 变更、交易状态的流转、外部系统的数据推送都是事件生产者。

**事件通道**：传递事件的中间件。我们使用了两种事件通道：Canal（监听 MySQL binlog）和 RocketMQ（传递业务事件）。

**事件消费者**：处理事件的组件。Redis 缓存同步、监控数据写入、外部系统通知都是事件消费者。

事件驱动架构的核心优势是松耦合和异步处理。生产者不关心谁消费事件，消费者不关心谁产生事件，两者通过事件通道解耦。

#### 三、项目落地实施

在项目中，我们实现了三个事件驱动场景：

**场景一：Canal 监听 MySQL 变更同步 Redis 缓存**

FiboRule 的规则配置、引擎节点、指标定义等数据存储在 MySQL 中，执行器通过 Redis 缓存读取这些配置。我们引入 Canal 组件，监听 MySQL 的 binlog 变更事件。当风控人员在管理端修改规则配置并保存到 MySQL 后，Canal 捕获到数据变更事件，自动将变更同步到 Redis 缓存。

Canal 的工作原理是伪装成 MySQL 从节点，通过主从复制协议获取 binlog 事件。我们配置 Canal 监听 `riskmanage` 数据库的关键表（`t_rule`、`t_rule_version`、`t_engine_node`、`t_field` 等），当这些表发生 INSERT、UPDATE、DELETE 操作时，Canal 解析 binlog 事件，提取变更数据，通过自定义的 Canal Client 将变更同步到 Redis。

关键设计点包括：一是幂等处理，同一条 binlog 可能被重复消费，我们通过版本号机制保证幂等性。二是顺序保证，Canal 保证同一张表的变更顺序，我们按表维度消费，避免乱序。三是异常处理，当 Redis 写入失败时，将事件写入重试队列，由后台线程重试。

**场景二：交易状态流转事件**

资金管理系统的交易生命周期包括经办（RG）、审批（AU）、确认（CK）、记账（AC/AK）等状态。我们使用工作流引擎驱动状态流转，每个状态变更都是一个事件。当经办人员提交交易后，系统产生"经办提交"事件，触发审批流程；审批通过后，产生"审批通过"事件，触发确认流程。

这种事件驱动的状态流转，使得每个状态的处理逻辑独立，新增状态只需新增事件处理器，无需修改已有流程。

**场景三：外部系统数据同步事件**

资金管理系统需要对接 CFETS 交易中心、中债托管系统等外部系统。我们通过消息队列解耦外部系统数据同步：外部系统推送数据变更事件到消息队列，资金管理系统消费事件并更新本地数据。这样即使外部系统暂时不可用，消息队列可以暂存事件，待外部系统恢复后继续消费。

#### 四、效果与反思

项目上线后，事件驱动架构的应用取得了以下效果：

一是规则配置变更到缓存生效的延迟从分钟级（手动刷新）降低到秒级（Canal 自动同步），消除了人为遗漏的风险。

二是系统间耦合度降低。资金管理系统不直接依赖 FiboRule 的可用性，通过事件队列缓冲，即使风控系统短暂不可用，交易系统也能继续运行（降级为不校验风控规则）。

三是可扩展性增强。新增事件消费者（如监控告警、数据分析）不影响已有系统。

不足之处在于：事件驱动增加了调试和排障的复杂度，需要完善的链路追踪和日志关联机制。Canal 的运维成本也需要关注，包括 binlog 格式配置、Canal Server 的高可用、消费延迟监控等。

## 五、论文报告：论系统安全架构设计及其应用

> 对应真题：2021年下半年-试题二
> 使用素材：FiboRule 接口鉴权与脚本沙箱（主）+ 紫金资金管理系统 token 鉴权（辅）

### 摘要

2021年，我参与了某银行金融市场部的资金管理系统及其风控平台（FiboRule）建设项目，担任系统架构师。针对金融交易系统面临的数据泄露、越权操作、脚本注入等安全威胁，我在项目中设计了多层次的安全架构。通过 token 鉴权机制保障用户身份可信，通过 RBAC 权限模型控制操作权限，通过脚本沙箱限制规则引擎的脚本执行能力，通过 SQL 参数化防止注入攻击。项目上线后，系统通过了银行信息安全审计，未发生安全事故。

### 正文

#### 一、项目背景

2021年，我参与了某银行金融市场部的资金管理系统及其风控平台（FiboRule）建设项目，担任系统架构师。资金管理系统覆盖债券、同业等全业务域，日均处理数百笔交易；FiboRule 支持动态规则配置和实时风控检测，包含 839 个 Java 文件。

金融交易系统面临多方面的安全威胁：一是用户身份冒充，攻击者可能伪造登录态访问系统；二是越权操作，普通用户可能访问或修改无权操作的规则和配置；三是脚本注入，FiboRule 支持 Groovy/Jython 脚本规则，恶意脚本可能执行系统命令或访问敏感资源；四是 SQL 注入，指标中心的 SQL 指标可能存在动态拼接风险。

#### 二、安全架构设计

我们设计了四层安全防护体系：

**认证层**：采用 JWT token 机制。用户登录后，服务端生成包含用户 ID、角色、机构信息的 JWT token，返回给前端。前端将 token 存储在 session storage 中，每次请求通过 `Authorization: Bearer {token}` 头部传递。服务端通过 token 过滤器校验 token 有效性和过期时间。

**鉴权层**：采用 RBAC 权限模型。用户通过角色关联资源，资源通过功能号（`funcid`）标识。每个接口调用前，权限切面校验当前用户的角色是否包含该功能号的操作权限。FiboRule 的规则配置、发布、查看等操作分别绑定不同的功能号，实现细粒度权限控制。

**脚本沙箱层**：FiboRule 支持 Groovy 和 Jython 脚本规则。我们通过 Java SecurityManager 和自定义沙箱限制脚本的执行能力：禁止访问文件系统、禁止执行系统命令、禁止网络访问、限制执行时间（超时自动终止）、限制内存使用。脚本只能访问白名单中的 API，如数学运算、字符串处理、日期计算等安全函数。

**数据安全层**：SQL 指标采用参数化查询，禁止动态拼接 SQL。敏感数据（如交易金额、授信额度）在日志中脱敏显示。数据库连接使用只读账号，防止误操作。

#### 三、效果与反思

项目上线后，系统通过了银行信息安全审计，未发生安全事故。token 鉴权和 RBAC 权限模型保障了用户身份和操作权限的可控性，脚本沙箱有效防止了恶意脚本的执行，SQL 参数化消除了注入风险。

不足之处在于：脚本沙箱的限制可能影响规则表达能力，部分复杂计算需要在沙箱外实现。JWT token 的注销机制需要额外设计（如黑名单表）。

## 六、论文报告：论性能测试

> 对应真题：2025年下半年-试题三
> 使用素材：FiboRule 执行器性能优化（主）+ 紫金资金管理系统报表性能（辅）

### 摘要

2021年，我参与了某银行金融市场部的风控规则引擎（FiboRule）建设项目，担任系统架构师。该系统作为资金管理系统的实时风控检测子系统，需要在交易高峰期间保持低延迟响应。针对系统上线前缺乏性能基线、执行器在高并发下延迟飙升的问题，我主导设计并实施了系统化的性能测试方案。通过负载测试确定系统承载能力，通过压力测试发现性能瓶颈，通过基准测试验证优化效果。项目上线后，单笔风控检测 P95 延迟从 800ms 优化到 180ms，系统承载能力从 50 TPS 提升到 300 TPS。

### 正文

#### 一、项目背景

2021年，我参与了某银行金融市场部的风控规则引擎（FiboRule）建设项目，担任系统架构师。FiboRule 采用管理端与执行器分离架构，执行器（ddp-runner-api）承接资金管理系统的实时风控调用，需要在 100-300ms 内完成单笔风险检测。

在项目开发完成后、上线前的测试阶段，我们发现执行器在并发超过 50 时，P95 刪迟从 100ms 飙升到 800ms 以上，无法满足业务要求。为定位性能瓶颈并建立性能基线，我主导设计并实施了系统化的性能测试方案。

#### 二、性能测试方案设计

我们设计了四类性能测试：

**负载测试**：逐步增加并发用户数（从 10 到 500），观察系统响应时间、吞吐量和资源利用率的变化趋势。目的是确定系统的最优并发数和最大承载能力。

**压力测试**：在超过预期负载（500 并发）的情况下持续运行，观察系统的降级行为和恢复能力。目的是验证系统在极端压力下的稳定性。

**基准测试**：在每次优化后重复相同的负载测试，对比优化前后的性能指标。目的是量化每次优化的效果。

**稳定性测试**：在预期负载（100 并发）下持续运行 24 小时，观察内存泄漏、连接池耗尽、缓存失效等长运行问题。

测试场景设计为模拟资金管理系统调用 `/runner/decision` 接口。测试数据准备了 1000 笔模拟交易，包含不同的交易类型、产品类型、交易对手和金额。测试工具使用 JMeter，监控工具使用 Grafana + Prometheus。

#### 三、性能瓶颈分析与优化

通过负载测试，我们发现了以下性能瓶颈：

**瓶颈一：数据库查询慢。** 规则执行过程中，SQL 指标需要查询外部数据库获取授信额度、持仓数据等。当并发增大时，数据库连接池耗尽，查询排队等待。优化措施：一是引入 Redis 缓存热点数据，减少数据库查询次数；二是优化 SQL 语句，添加索引；三是增大 Druid 连接池大小并配置连接回收策略。

**瓶颈二：外部接口延迟高。** 规则执行链中的 RPC 节点需要调用外部行情、评级接口。外部接口响应时间不稳定，偶发高延迟会阻塞整个决策流。优化措施：一是设置超时控制（300ms），超时后返回默认值；二是引入熔断器（Hystrix），当外部接口错误率超过阈值时自动熔断；三是异步化非关键指标的获取。

**瓶颈三：规则解析开销大。** 每次执行时，执行器需要从数据库加载规则配置并解析。在高并发下，规则解析成为 CPU 瓶颈。优化措施：一是将规则配置缓存到 Redis，减少数据库访问；二是引入规则预编译，将解析结果缓存到本地内存；三是 Canal 监听 MySQL 变更自动同步缓存，保证缓存新鲜度。

**瓶颈四：串行执行效率低。** 决策流中的多个节点串行执行，当节点数量多时，总延迟累加。优化措施：一是对无依赖关系的节点引入并行执行（使用线程池）；二是对耗时指标的获取引入异步预取。

优化后，我们通过基准测试验证效果：单笔风控检测 P95 延迟从 800ms 优化到 180ms，系统承载能力从 50 TPS 提升到 300 TPS。

#### 四、效果与反思

项目上线后，性能测试方案持续发挥作用。每次版本发布前，我们都会执行基准测试，确保新版本不引入性能退化。稳定性测试帮助我们发现了一个 Redis 连接泄漏的 Bug，在上线前及时修复。

不足之处在于：性能测试数据是模拟数据，与真实交易数据的分布存在差异；外部接口的 Mock 不够真实，无法完全模拟外部系统的延迟特征。后续我们计划引入生产流量回放（Traffic Replay）技术，使用真实流量进行性能测试。

## 七、论文报告：论秒杀场景及其技术解决方案

> 对应真题：2025年下半年-试题四
> 使用素材：FiboRule 批量执行与限流（主）+ 紫金资金管理系统月末高峰（辅）

### 摘要

2021年，我参与了某银行金融市场部的资金管理系统及其风控平台（FiboRule）建设项目，担任系统架构师。资金管理系统在月末、季末出现交易高峰，FiboRule 需要在短时间内处理大量风控检测请求，具有典型的高并发、短时效特征。针对瞬时高并发场景下的系统稳定性问题，我在项目中引入了限流、缓存、熔断、异步化等技术方案。通过令牌桶算法在网关层限流，通过 Redis 缓存减少数据库访问，通过熔断器保护外部接口，通过线程池实现批量执行的并发处理。项目上线后，系统在月末交易高峰期间稳定运行，未出现超时和服务雪崩。

### 正文

#### 一、项目背景

2021年，我参与了某银行金融市场部的资金管理系统及其风控平台（FiboRule）建设项目，担任系统架构师。资金管理系统覆盖债券、同业等全业务域，FiboRule 作为其交易风险检测子系统，需要对每笔交易进行实时风控检测。

在月末和季末，资金管理系统出现交易高峰，交易量是平时的 3-5 倍。FiboRule 的执行器需要在短时间内处理大量风控检测请求，具有瞬时高并发、业务逻辑集中、数据一致性要求严格的特点。如果系统承载能力不足，会导致响应超时、服务降级甚至雪崩。

#### 二、高并发技术方案设计

针对高并发场景，我们采用了"限流-缓存-熔断-异步化"的协同方案：

**限流**：在网关层（Nginx）配置令牌桶限流，限制每秒请求数不超过执行器的最大承载能力（300 QPS）。超过限流阈值的请求直接返回"系统繁忙，请稍后重试"，避免压垮后端服务。在应用层，使用信号量限制同时执行的决策流数量，防止线程池耗尽。

**缓存**：将规则配置、引擎节点、指标定义等热点数据缓存到 Redis。执行器每次执行时，优先从 Redis 读取配置，缓存未命中时才查询数据库。通过 Canal 监听 MySQL 变更自动同步缓存，保证缓存新鲜度。缓存命中率从优化前的 60% 提升到 95%。

**熔断**：对外部接口（行情、评级、授信等）配置熔断器。当外部接口错误率超过 50% 或响应时间超过 500ms 时，自动熔断，返回默认值（如"无法获取行情，建议人工复核"）。熔断器每隔 10 秒尝试半开，探测外部接口是否恢复。

**异步化**：批量风控执行使用受控线程池并发处理。线程池核心线程数设置为 CPU 核心数的 2 倍，最大线程数设置为 CPU 核心数的 4 倍，队列长度限制为 1000。超过队列容量的请求直接拒绝，返回"批量任务已满，请稍后重试"。

#### 三、效果与反思

项目上线后，系统在月末交易高峰期间稳定运行。限流机制有效防止了瞬时高并发压垮后端服务，缓存机制将数据库访问量降低了 80%，熔断机制保护了外部接口故障时的系统稳定性，异步化机制将批量执行效率提升了 3 倍。

不足之处在于：限流阈值需要根据实际负载动态调整，当前为静态配置；熔断器的半开探测策略可能导致短暂的请求失败。后续我们计划引入自适应限流（根据实时负载自动调整阈值）和更精细的熔断策略。

## 八、论文报告：论微服务架构及其应用

> 对应真题：2021年下半年-试题四
> 使用素材：紫金资金管理系统 qiankun 微前端（主）+ FiboRule 服务拆分（辅）

### 摘要

2021年，我参与了某银行金融市场部的资金管理系统建设项目，担任系统架构师。该系统覆盖债券、同业、外汇、存单等全业务域，原有单体前端架构导致构建慢、发布耦合、团队协作困难。我在项目中引入微服务架构思想，采用 qiankun 微前端方案将系统拆分为 14 个子应用，通过统一门户接入、跨应用状态同步、公共能力下沉等机制实现独立开发和独立发布。项目上线后，单应用构建时间从 15 分钟降低到 3 分钟，团队并行开发效率显著提升。

### 正文

#### 一、项目背景

2021年，我参与了某银行金融市场部的资金管理系统建设项目，担任系统架构师。该系统面向银行资金业务部门，覆盖债券投资、同业拆借、外汇资金、同业存单、现金头寸、非标投资、标准化票据等全业务域。

在项目初期，所有业务域在一个前端工程中开发。随着业务域持续扩展，单体架构的问题日益突出：构建时间长达 15 分钟，一个业务域的修改需要整体回归测试，多个团队在同一代码库中协作冲突频繁，发布时所有业务域必须同步上线。

#### 二、微服务架构设计

我们采用 qiankun 微前端方案，将系统拆分为 14 个子应用：

- **主应用（master）**：负责登录、菜单、顶部/底部导航、任务列表、页签、子应用注册和跨应用状态同步
- **业务子应用**：balm-bond（债券）、balm-interbank（同业）、balm-currency（外汇）、balm-interbank-receipt（存单）、balm-cash-management（现金头寸）、balm-nonStandardized-investment（非标）、balm-standardized-bills（票据）等
- **支撑子应用**：balm-system（系统管理）、balm-middleoffice-management（中台管理）、balm-parameter-management（参数管理）、balm-inquire-statistics（报表查询）、balm-market（市场行情）

主应用通过 `micro-app.js` 注册子应用，开发环境 entry 为 `//localhost:{port}`，生产环境 entry 为 `/{appName}`。子应用通过 `bootstrap/mount/unmount` 生命周期接入，在 qiankun 环境下用主应用下发的 `routerBase` 作为 VueRouter base。

跨应用状态通过 `initGlobalState` 实现。主应用初始化 `userInfo`、`systemInfo`、`token` 等全局状态，子应用通过 `onGlobalStateChange` 监听变化。典型场景包括：主应用广播系统日子应用关闭页签时通过 `setGlobalState` 通知主应用。

公共能力通过两个内部包下沉：`web-frame` 提供登录态、HTTP、字典、日期金额工具等基础能力；`web-components` 提供工作流、记账弹窗、打印、附件等业务组件。各子应用通过 `npm run pull` 从内部私库拉取公共包。

#### 三、效果与反思

项目上线后，微服务架构的效果显著：单应用构建时间从 15 分钟降低到 3 分钟；债券、同业、报表等团队可独立开发和发布，互不影响；新增业务域只需新建子应用并注册到主应用即可。

不足之处在于：跨应用状态同步存在延迟，子应用间的公共依赖需要严格版本管理，qiankun 的 CSS 隔离不完全，需要手动处理样式冲突。
