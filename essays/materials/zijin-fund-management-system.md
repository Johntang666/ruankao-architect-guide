## 一、项目背景（一句话版）

> 我参与的主力项目是某银行金融市场部的紫金资金管理系统，该系统面向银行资金业务部门，覆盖债券、同业拆借、外汇/外币资金、同业存单、现金头寸、非标投资、标准化票据等全业务域，采用 qiankun 微前端架构，支撑日均数百笔交易的经办、审批、确认、记账和报表输出。

## 二、项目背景（详细版，可直接用于论文摘要）

紫金资金管理系统是某银行金融市场部的核心业务系统，面向银行资金业务部门，覆盖债券投资（现券买卖、回购、承销、融入融出）、同业拆借、外汇/外币资金、同业存单（投标、发行、买卖、兑付）、现金头寸管理、非标准化同业投资、标准化票据等全业务域。系统采用一个主应用聚合多个业务子应用的微前端模式，用户从统一门户进入菜单和待办任务，再按业务类型进入具体经办、审批、确认、记账、查询和报表页面。系统日均处理数百笔交易，对接 CFETS 交易中心、中债托管系统、核心账务系统、ESB 总线等十余个外部系统。

## 五、原架构问题与改造动机

### 改造前的问题

| 问题 | 表现 |
|---|---|
| 单体应用体积庞大 | 所有业务域在一个前端工程中，构建慢、冲突多、发布耦合 |
| 业务规则分散在页面 | 授信、风险暴露、关联交易、价格偏离等规则分布在各 Vue 组件中，难维护、难测试 |
| 公共逻辑重复实现 | 登录态、HTTP、工作流、记账、打印等能力在各业务页面重复编码 |
| 外部系统对接散乱 | CFETS、中债、核心等对接逻辑散布在各业务页面，缺少统一集成层 |
| 报表与交易耦合 | 报表查询和交易经办混在同一个应用中，影响性能和发布节奏 |

### 改造动机

- 业务域持续扩展（从债券扩展到同业、外汇、存单、非标、票据），需要支持团队并行开发
- 监管要求趋严，报表和审计需求增加，需要独立的报表能力
- 外部系统对接越来越多，需要统一集成架构
- 银行内网部署约束，需要桌面化交付方案

## 六、架构选型依据

| 决策点 | 选择 | 为什么选 | 权衡/代价 |
|---|---|---|---|
| 前端架构 | qiankun 微前端 | 支持多业务域并行开发、独立构建、统一门户接入 | 跨应用状态、公共依赖、路由、版本治理复杂 |
| 后端接入 | 统一交易网关 `/cmsmessage + prcscd` | 适合银行交易系统的处理码分发和历史系统接入 | 接口语义不直观，契约治理依赖处理码文档 |
| 工作流 | 公共 mixin 承载 | 快速复用经办、审批、确认、记账等动作 | mixin 隐式依赖强，状态机不显性 |
| 本地承载 | Koa/Electron | 解决内网桌面化、跨域、静态资源托管、自动更新 | 引入本地服务端口、配置、更新治理成本 |
| 外部集成 | ESB 总线 + 直联 | 统一对接交易中心、托管、核心、名单平台等外部系统 | ESB 成为单点依赖，需要降级和补偿机制 |
| 报表架构 | 独立报表子应用 + 动态表头 | 报表口径独立演进，不影响交易主流程 | 报表口径、模板版本需要独立治理 |

## 七、关键技术点清单

### 7.1 qiankun 微前端架构

- 主应用 `master` 负责登录、菜单、顶部/底部导航、任务列表、页签、子应用注册和跨应用状态同步
- 子应用通过 `bootstrap/mount/unmount` 生命周期接入，开发环境用 `localhost:{port}`，生产环境用 `/{appName}`
- 跨应用状态通过 `initGlobalState` 广播 `userInfo/systemInfo/token/taskToken`
- 子应用激活规则为 `/#/{appName}`，容器为 `#microContainer`

### 7.2 统一交易网关与处理码分发

- 所有业务 POST 默认打到 `/api/cmsmessage`
- `prcscd`（处理码）作为后端服务路由键，`funcid_` 作为菜单/功能权限和审计口径
- `bktrtp`（业务类型）+ `taskst`（流程节点）+ `bktrsq`（业务流水）构成统一业务模型
- 后端至少包含：交易域服务、规则与风控服务、会计记账服务、报表打印服务、系统参数服务、外部集成服务

### 7.3 交易生命周期状态机

| 状态码 | 含义 | 动作 |
|---|---|---|
| RG | 经办 | 录入/修改/提交 |
| AU | 审批 | 同意/不同意/打回 |
| CK | 确认 | 交易确认/指令确认 |
| AC | 记账准备 | 获取会计分录、校验结算金额 |
| AK | 记账通过 | 完成记账、生成传票 |
| MD | 市场匹配 | CFETS/CSTP 下行数据匹配 |
| QS | 后台投标 | 后台投标确认 |

- 公共工作流方法：`RG` 经办提交、`AU` 审批同意、`disagree` 不同意打回、`CK` 确认、`delTask` 终止、`nextNode` 强制流转
- `succtg` 处理结果：`S` 同意/提交/通过，`F` 不同意/更正/终止

### 7.4 公共能力抽象

| 能力 | 实现 | 复用范围 |
|---|---|---|
| 登录态管理 | token + session storage + 路由守卫 | 全部子应用 |
| HTTP 封装 | axios 拦截器 + 自动绑定用户/系统日/funcid | 全部业务请求 |
| 工作流 | `workflow.js` mixin | 所有交易页面 |
| 记账弹窗 | `AcctioDialog.vue` | 所有需要记账的业务 |
| 打印能力 | hiprint + 打印 mixin | 审批单、传票、成交单 |
| 金额日期工具 | `web-frame/utils` | 全部业务页面 |
| 字典数据 | `web-frame/utils/model` | 全部业务页面 |

### 7.5 外部系统集成

| 外部系统 | 集成方式 | 业务场景 |
|---|---|---|
| CFETS/CSTP 交易中心 | 下行数据匹配 + 联机交易 | 现券、回购、信用拆借成交明细匹配 |
| 中债/直联系统 | 直联数据校验 | 债券交易要素核验、托管账户匹配 |
| 核心账务系统 | 会计分录 + 记账 | 交易记账、传票生成、核心流水对账 |
| ESB 总线 | 统一服务调用 | 大额风险暴露查询、关联交易查询、统一名单平台 |
| ODS/报表数据集市 | 数据导出 | 监管报表、经营报表、统计分析 |

### 7.6 报表与打印中心

- 独立子应用 `balm-inquire-statistics`，163 个路由
- 动态表头：`Reporter.getReportTitle` + `Reporter.execReport`
- 报表分类：债券报表、货币市场报表、同业报表、NCD 报表、投资收益、司库及自营、风险管理与部门考核、核心对账、监管报表
- 打印中心：`hiprint` 组件 + 打印 mixin，支持审批单、传票、成交单、无纸化凭证

### 7.7 Koa/Electron 本地承载

- Koa 本地服务：`/api` 反向代理到真实后端、静态资源托管
- Electron 壳：桌面窗口、IPC 通信、禁用 F5、版本更新、日志清理
- 解决银行内网部署约束：隐藏后端地址差异、跨域、桌面化分发

## 八、实施难点与解决办法

| 难点 | 解决办法 |
|---|---|
| 多应用版本不一致 | 统一 Vue 版本基线，公共包语义化版本管理，兼容性测试 |
| 跨应用状态同步 | qiankun globalState 广播 userInfo/systemInfo/token，子应用监听变化 |
| 公共包传播治理 | 内部 npm 私库发布，`npm run pull` 统一拉取，变更日志和兼容测试 |
| 交易前校验规则分散 | 前端做体验型预校验，强规则后端裁决，统一规则版本 |
| 外部系统对接复杂 | ESB 统一接入层，CFETS/CSTP 下行匹配服务，直联数据校验服务 |
| 报表口径多样 | 独立报表子应用，动态表头，报表元数据管理 |
| 银行内网部署约束 | Koa 本地代理 + Electron 桌面壳 + SFTP 部署 |

## 十、架构设计亮点（可直接用于论文）

1. **按业务域拆分微前端**：债券、同业、外汇、现金、参数、系统、报表等独立开发、独立构建、统一接入，支持团队并行。
2. **统一业务模型**：用 `bktrtp + funcid + taskst + bktrsq` 将菜单、任务、交易、审批、记账、打印串成统一流水线。
3. **公共能力下沉**：工作流、记账、打印、HTTP、字典等公共能力从业务页面中抽离到 `web-frame` 和 `web-components`，降低重复实现成本。
4. **统一交易网关**：采用 `/cmsmessage + prcscd` 的处理码分发模式，适合银行交易系统的契约管理和历史系统接入。
5. **Koa/Electron 适配银行内网**：通过本地代理解决跨域和后端地址差异，通过 Electron 壳实现桌面化部署和自动更新。
6. **报表与交易分离**：报表中心独立演进，动态表头支撑多口径报表，打印中心统一输出审批单、传票和凭证。
