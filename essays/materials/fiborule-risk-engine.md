# 母项目素材卡：FiboRule 风控规则引擎

> 用途：论文写作母项目素材，可按不同主题抽取角度复用。
> 定位：附加项目，偏平台/中台系统，覆盖规则引擎、决策流、指标中心、监控审计。
> 搭配系统：紫金资金管理系统（主系统）。

---

## 一、项目背景（一句话版）

> 在资金管理系统基础上，为解决交易风控规则频繁变化导致主系统发版频繁的问题，我主导设计了 FiboRule 风控规则引擎，将风控策略从交易系统中剥离为独立平台，支持规则动态配置、可视化决策流编排、多策略融合执行、版本管理和审计追溯。

## 二、项目背景（详细版）

FiboRule 是一个实时智能决策引擎和规则引擎，定位为资金管理系统的交易风险检测子系统。资金管理系统在交易录入、修改、审批、确认前，将交易要素、交易对手、产品、组合、授信、行情、评级等指标传入 FiboRule；FiboRule 根据已发布的决策流执行名单、规则、评分卡、决策表、决策树、模型和外部接口调用，返回放行、提示、复核或拒绝结果。核心理念是把高频变化的风险政策抽象成可配置、可版本化、可审计的策略资产，而不是写死在交易系统代码中。

## 五、原架构问题与改造动机

### 改造前：规则硬编码在资金管理系统中

| 问题 | 后果 |
|---|---|
| 每次规则变化都要改交易系统代码 | 发版频繁，回归范围大，影响主交易流程稳定性 |
| 风控人员无法直接维护规则 | 技术团队成为规则变更瓶颈 |
| 规则命中过程不可解释 | 审计时只能看最终结果，难以说明哪条规则命中 |
| 多业务复用困难 | 债券、同业存单、存款、回购等规则难以统一沉淀 |
| 灰度和回滚困难 | 规则变更无法像策略资产一样管理版本 |

### 改造动机

- 风控规则变化频率远高于交易主流程（每月数次 vs 季度级别）
- 监管要求风险结果必须可解释、可审计
- 多业务线需要复用同一套风控能力
- 需要支持风控人员自助配置，减少对技术团队的依赖

## 六、架构选型依据

| 决策点 | 选择 | 为什么选 | 权衡/代价 |
|---|---|---|---|
| 规则引擎独立部署 | 从资金系统剥离为独立服务 | 交易系统保持稳定，规则变化在风控平台内完成 | 多一个系统边界，需要接口、监控和容灾治理 |
| 多策略并存 | 规则、评分卡、决策表、决策树、名单库、模型等 | 不同风控问题适合不同策略表达方式 | 配置治理成本上升，业务人员需要规范建模 |
| 决策流引擎 | 可视化节点编排 | 多策略需要按流程编排，有前后依赖关系 | 需要校验 DAG、版本和运行路径 |
| 管理端与执行器分离 | 双 Spring Boot 启动模块 | 管理关注配置和权限，执行关注低延迟和高可用 | 需要处理配置发布、缓存同步和版本一致性 |
| 指标中心 | 统一抽象数据获取 | 规则需要的数据来源复杂（入参/SQL/接口/常量/衍生） | 指标中心成为关键基础设施，需要治理 |
| 版本与快照 | 每次发布保存配置快照 | 风控必须可审计、可追溯、可回滚 | 存储量和日志治理压力增加 |

## 七、关键技术点清单

### 7.1 四大中心架构

| 设计中心 | 解决的问题 | 核心能力 |
|---|---|---|
| 数据中心（DataX） | 规则运行需要哪些数据、数据从哪里来 | 基础指标、常量指标、SQL 指标、接口指标、衍生指标、实时指标 |
| 策略中心（StrategyX） | 规则如何配置和版本化 | 复杂规则、脚本规则、评分卡、决策表、决策树、名单库、模型 |
| 引擎中心（EngineX） | 多策略如何编排成完整检测流程 | 可视化决策流、节点递归执行、并行/聚合/分组/冠军挑战 |
| 监控中心（Monitor） | 每次执行命中什么、为什么 | 结果集、决策流监控、节点监控、策略监控 |

### 7.2 指标中心设计

指标是规则运行时的最小业务数据单元。获取优先级：
1. 调用入参（交易系统直接传入）
2. 常量指标（固定阈值、枚举）
3. SQL 指标（从外部数据库查询）
4. 接口指标（调用外部 HTTP 接口）
5. 衍生指标（由已有指标计算，如收益率偏离、集中度）

资金管理场景典型指标：交易金额、交易方向、债券代码、发行人、主体评级、债项评级、剩余期限、久期、收益率、估值收益率、交易对手、交易员、组合、账户、授信占用、当日累计交易额、集中度、内部名单状态、市场行情偏离。

### 7.3 多策略融合

| 策略类型 | 适合表达 | 资金管理例子 |
|---|---|---|
| 复杂规则 | 多层条件、嵌套、命中输出 | 授信余额不足、交易员权限不足、组合集中度超限 |
| 脚本规则 | 公式复杂、需要自定义计算 | 特殊债券估值偏离、结构化参数计算 |
| 评分卡 | 多维度加权评分 | 交易风险综合分、交易对手风险评分 |
| 决策表 | 制度矩阵 | 评级 + 期限 + 金额映射审批级别 |
| 决策树 | 分层路径判断 | 产品类型 → 评级 → 期限 → 处置动作 |
| 名单库 | 集合命中 | 交易对手黑名单、禁投发行人名单 |
| 模型 | 机器学习预测 | 违约概率、异常交易概率 |
| RPC 节点 | 外部服务调用 | 查询行情、评级、额度占用 |

### 7.4 决策流引擎

决策流由节点和连线组成，从 `ND_START` 开始按 `next_nodes` 递归执行：

| 节点类型 | 作用 |
|---|---|
| 开始节点 | 决策流入口 |
| 规则集节点 | 执行复杂规则和脚本规则 |
| 分组节点 | 根据条件选择下一节点 |
| 评分卡节点 | 多维评分并输出分数 |
| 名单库节点 | 黑白名单命中判断 |
| 决策表节点 | 行列条件定位结果 |
| 决策树节点 | 树状路径判断 |
| 模型节点 | 加载 PMML 模型预测 |
| RPC 节点 | 调用外部接口 |
| 并行节点 | 多分支并行执行 |
| 聚合节点 | 并行分支汇聚 |
| 冠军挑战节点 | 选择冠军分支执行 |
| 子引擎节点 | 调用另一个引擎 |

### 7.5 版本与审计

| 机制 | 作用 |
|---|---|
| `t_engine_version.boot_state` | 控制当前运行版本 |
| 策略版本表 | 保存规则、评分卡、决策表、决策树等版本 |
| `snapshot` 字段 | 保存配置快照，供监控和审计 |
| `t_resultset` | 保存每次执行输入输出 |
| 监控表 | 保存节点和策略执行过程 |

核心审计能力：能回答"某一笔交易当时执行的是哪一版规则""那一版规则当时具体配置是什么""后续规则变更后历史交易结果是否还能解释"。

### 7.6 管理端与执行器分离

| 模块 | 默认端口 | 职责 | 流量特征 |
|---|---|---|---|
| ddp-manager-web | 8080 | 规则配置、指标配置、引擎配置、权限和监控查询 | 低频写入和查询 |
| ddp-runner-api | 8081 | 对资金管理系统提供实时决策接口 | 高频实时调用 |

分离原因：
1. 管理端故障不影响已发布规则的实时执行
2. 执行器可单独扩容，承接交易高峰
3. 管理操作和交易执行可分别做安全隔离

### 7.7 缓存与数据同步

| 组件 | 设计原因 |
|---|---|
| Redis | 缓存引擎、节点、规则、字段等配置，提高执行器读取速度 |
| Canal | 监听 MySQL 变更并同步缓存，减少人工刷新缓存 |
| HBase | 存储大规模监控明细，适合高频执行日志和节点明细 |

### 7.8 典型风控规则样例

| 规则名称 | 规则表达 | 策略类型 | 输出 |
|---|---|---|---|
| 交易对手黑名单拦截 | counterparty_id 命中黑名单 | 名单库 | REJECT |
| 授信余额不足 | trade_amount + used_credit > credit_limit | SQL 指标 + 复杂规则 | REJECT |
| 估值偏离预警 | abs(yield_rate - valuation_yield) > threshold | 接口指标 + 复杂规则 | REVIEW |
| 债项评级不足 | issuer_rating < AA | 决策表 | REJECT |
| 非授权交易品种 | 交易员权限不包含产品类型 | 复杂规则 | REVIEW |
| 组合久期超限 | duration_after_trade > duration_limit | SQL 指标 + 评分卡 | REVIEW |
| 单一发行人集中度 | issuer_position / portfolio_nav > 0.1 | SQL 指标 + 复杂规则 | WARN/REVIEW |

### 7.9 推荐的交易风险引擎编排

1. 开始节点
2. RPC 节点：获取外部行情、评级、估值收益率
3. 名单库节点：检查交易对手黑名单、发行人禁投名单
4. 规则集节点：校验交易员权限、产品准入、期限限制
5. SQL 指标 + 规则集：校验授信额度、单一发行人集中度、组合集中度
6. 评分卡节点：计算交易风险综合分
7. 决策表节点：根据风险分、交易金额、评级、偏离度映射处置动作
8. 决策选项节点：输出 PASS/WARN/REVIEW/REJECT
9. 监控和结果记录

## 八、实施难点与解决办法

| 难点 | 解决办法 |
|---|---|
| 规则动态化带来脚本安全风险 | 引入脚本沙箱、白名单 API、执行超时、内存限制和审计 |
| SQL 指标存在动态 SQL 拼接风险 | 参数白名单、预编译、SQL 审核、只读账号、超时和限流 |
| 外部指标影响实时性能 | 缓存、超时、重试、熔断、降级、备用数据 |

| 并行节点共享变量池的并发问题 | 分支上下文复制，聚合节点显式合并，避免并发写冲突 |
| 规则配置错误影响业务 | 规则测试样例、发布审批流程、版本快速回滚 |
| 版本切换的一致性保障 | 发布时版本原子切换、失败自动回滚、保留上一运行版本 |

## 十、架构设计亮点（可直接用于论文）

1. **规则外置**：以业务变化率为边界拆分系统，变化慢的交易流程留在主系统，变化快的风控规则沉淀为平台能力。
2. **指标中心解耦**：规则只引用指标，不关心数据来源，实现数据获取与规则表达的分离。
3. **多策略融合**：针对不同风控问题选择最合适的策略表达方式，而不是把所有业务塞进一种规则里。
4. **决策流编排**：把风控执行过程显式建模为可视化决策流，而非隐藏在代码调用顺序中。
5. **版本与快照**：每次发布保存完整配置快照，支持审计追溯"当时用哪一版、命中了什么、为什么拒绝"。
6. **管理端与执行器分离**：读写流量性质不同，分别部署、分别扩容、分别做安全隔离。
7. **监控闭环**：结果集 + 节点监控 + 策略监控，支持风控结果的完整解释和复盘。

### 简短版（1分钟）

> 在资金管理系统基础上，我主导设计了 FiboRule 风控规则引擎。该系统将风控策略从交易系统中剥离为独立平台，采用"指标中心 + 策略中心 + 引擎中心 + 监控中心"的四中心架构。指标中心统一抽象数据获取，策略中心支持规则、评分卡、决策表、决策树、名单库、模型等 8 种策略类型，引擎中心通过可视化决策流编排多策略执行，监控中心记录完整的执行过程和结果。管理端与执行器分离部署，支持独立扩容和安全隔离。系统通过版本快照和结果集实现风控审计追溯。

### 完整版（3分钟）

> 在资金管理系统的基础上，为解决交易风控规则频繁变化导致主系统发版频繁的问题，我主导设计了 FiboRule 风控规则引擎。
>
> 该系统的核心设计理念是"规则外置"：以业务变化率为边界拆分系统，变化慢的交易流程留在主系统，变化快的风控规则沉淀为平台能力。
>
> 架构上采用四中心设计：数据中心通过指标抽象统一管理规则运行所需数据，支持从入参、SQL、接口、常量和衍生公式中获取；策略中心支持复杂规则、脚本规则、评分卡、决策表、决策树、名单库、机器学习模型等 8 种策略类型，让风控人员用合适的工具描述风险政策；引擎中心通过可视化决策流将多个策略编排成完整检测流程，支持分支、并行、聚合、子引擎等节点类型；监控中心记录每次执行的完整过程，支持审计追溯。
>
> 部署上采用管理端与执行器分离模式：管理端负责规则配置和权限管理，执行器负责实时决策调用。两者读写流量性质不同，可分别部署、分别扩容、分别做安全隔离。
>
> 系统通过版本和快照机制解决审计问题：每次规则发布保存完整配置快照，结合结果集记录，能回答"某一笔交易当时执行的是哪一版规则""命中了什么""为什么拒绝"。
>
> 该系统的主要架构权衡在于：规则动态化提升了灵活性，但带来脚本安全和发布一致性问题；指标中心增强了数据获取能力，但外部指标会影响实时性能；可视化决策流便于业务理解，但复杂流程需要控制节点数量。我们通过脚本沙箱、缓存预热、超时熔断、版本原子切换等机制逐一应对。
